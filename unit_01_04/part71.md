# 디자인 패턴

* 디자인 패턴의 3가지 케이스
-> 생성, 구조, 행위

## 생성 패턴
* 구체적인 클래스에 의존하지 않고, 인터페이스를 통해
서로 의존함.
-> 추상 팩토리

* 작게 분리된 인스턴스를 건축 하듯이 조합하여 객체를 생성함
-> 빌더

* 객체 생성을 서브 클래스에서 처리하도록 분리하여 캡슐화한 패턴
-> 팩토리 메소드

* 원본 객체를 복제하는 방법으로 객체 생성하는 패턴
-> 프로토타입

* 하나의 객체를 생성하면 생성된 객체를 어디서든 참조할 수
있지만, 여러 프로세스가 동시에 참조할 순 없음
-> 싱글톤 

## 구조 패턴
* 호환성이 없는 클래스들의 인터페이스들을 다른 클래스가 이용
할 수 있도록 변환
-> 어댑터

* 구현부에서 추상층을 분리하여, 서로가 독립적으로 확장할 수
있도록 구성한 패턴
-> 브리지

* 여러 객체를 가진 복합 객체와 단일 객체를 구분 없이
다루고자 할 때 사용하는 패턴
-> 컴포지트

* 객체 간의 결합을 통해 능동적으로 기능들을 확장할 
수 있는 패턴
-> 데코레이터

* 복잡한 서브 클래스들을 피해 더 상위에 인터페이스를 구현
-> 파싸드

* 인스턴스를 가능한 공유
-> 플라이웨이트

* 접근이 어려운 객체와 여기에 연결하려는 객체 사이의
인터페이스 역할을 수행
-> 프록시

## 행위패턴
* 요청을 처리할 수 있는 객체가 둘 이상 존재.
* 하나가 처리 못하면 다음 객체로 넘어감
-> 책임 연쇄

* 요청을 객체의 형태로 캡술화해 재이용/취소할 수 있도록
정보를 저장 및 로그 형태로 남기는 패턴
-> 커맨드

* 언어의 문법 표현을 정의하는 패턴
-> 인터프리터

* 접근이 잦은 객체에 동일한 인터페이스를 사용
-> 반복자

* 수많은 객체들 간 복잡한 상호작용을 캡슐화해
객체로 정의하는 패턴
-> 중재자

* 특정 시점에서 객체 내부 상태를 객체화. 
-> 메멘토

* 한 객체의 상태가 변화하면 객체에 상속된 다른 객체에게
변화된 상태를 전달하는 패턴
-> 옵서버

* 객체의 상태에 따라 동일한 동작을 다르게 처리해야 
할 때 사용하는 패턴
-> 상태

* 동일한 계열의 알고리즘을 개별적으로 캡슐화하여
상호 교환할 수 있게 정의하는 패턴
-> 전략

* 상위 클래스에서 골격 정의, 하위 클래스에서 
세부 처리를 구체화
-> 템플릿 메소드

* 각 클래스의 데이터 구조에서 처리 기능을 분리해
별도의 클래스로 구성
-> 방문자
